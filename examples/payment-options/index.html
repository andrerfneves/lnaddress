LUD-XX Prototype

One Address. Every Rail.

A single Lightning Address becomes a universal router for Bitcoin payments. No separate endpoints. No broken compatibility. Just pick the rail that fits.

The Problem

Today, if a service wants to accept Bitcoin over multiple rails, it needs separate endpoints for each one. The wallet has to know the rail before it resolves.

Before (today)

  • One endpoint for Lightning
  • One endpoint for BOLT12
  • One endpoint for onchain
  • One endpoint for Liquid
  • Wallet must know rail before resolve
  • No unified verification

After (LUD-XX)

  • One endpoint, one flow
  • Discover rails at resolve time
  • Same UX, different destination
  • LUD-21 verify works for all rails
  • Legacy wallets ignore, still work
  • Modern wallets pick the best rail

🎯 The Big Idea

alice@example.com resolves to a single payRequest. That response contains a paymentOptions array. The wallet sees the available rails, the user picks one, and the same callback returns the appropriate destination — BOLT11 invoice, BOLT12 offer, onchain address, or Liquid address. No new discovery endpoint. No breaking change to existing wallets.

1
Resolve the Lightning Address
GET

The wallet resolves alice@example.com. The provider returns a standard LUD-06 payRequest plus the new paymentOptions array.

// GET https://example.com/.well-known/lnurlp/alice // Response: { "tag": "payRequest", "callback": "https://example.com/callback", "minSendable": 1000, "maxSendable": 100000, "metadata": "[[\"text/plain\",\"Pay alice@example.com\"]]", "paymentOptions": [ { "id": "lightning", "type": "lightning", "available": true }, { "id": "lightning-bolt12", "type": "lightning-bolt12", "available": true }, { "id": "onchain", "type": "onchain", "available": true }, { "id": "liquid", "type": "liquid", "available": true }, { "id": "arkade", "type": "arkade", "available": false }, { "id": "spark", "type": "spark", "available": true } ] }
import { resolve } from "lnaddress"; const payRequest = await resolve("alice@example.com"); console.log(payRequest.paymentOptions); // [{ id: "lightning", type: "lightning" }, ...]

3
Request the Payment Destination
GET

The wallet calls the callback with amount and paymentOption. The provider returns the destination for the selected rail.

// GET https://example.com/callback?amount=25000&paymentOption=liquid // Response: { "status": "OK", "paymentOption": "liquid", "paymentDestination": "lq1qq2x6t...", "paymentUri": "liquidnetwork:lq1qq2x6t...?amount=0.001", "verify": "https://example.com/verify/pay_123" }
import { requestPayment } from "lnaddress"; const payment = await requestPayment(payRequest, { amountMsat: 25_000, paymentOption: "liquid", }); // payment.type === "destination" // payment.paymentOption === "liquid" // payment.paymentDestination === "lq1qq2x6t..." // payment.paymentUri === "liquidnetwork:lq1qq2x6t...?amount=0.001"
↓
4
Verify Settlement (LUD-21)
GET

The wallet polls the verify URL. The same LUD-21 semantics now work for non-BOLT11 rails. The response includes the payment reference (txid, hash, etc.) once the service observes it.

// GET https://example.com/verify/pay_123 // Response — payment not yet seen: { "status": "OK", "settled": false, "paymentOption": "liquid", "paymentDestination": "lq1qq2x6t...", "paymentReference": null } // Response — payment confirmed: { "status": "OK", "settled": true, "paymentOption": "liquid", "paymentDestination": "lq1qq2x6t...", "paymentReference": "abc123txid" }
import { verifyPayment } from "lnaddress"; const result = await verifyPayment(payment); // result.paymentOption === "liquid" // result.paymentReference === "abc123txid"

🚀 Why This Is Transformational

Before this LUD, Bitcoin rails were fragmented. Lightning, BOLT12, onchain, Liquid — each required its own plumbing. Wallets had to know the rail before talking to the user.

With paymentOptions, the LNURL-pay endpoint becomes the router. The user resolves one address. The provider says what's available. The user picks. The same callback, the same verify, the same flow. One protocol, every rail.

Legacy wallets ignore paymentOptions and keep working. Modern wallets get superpowers. That's the whole point.
